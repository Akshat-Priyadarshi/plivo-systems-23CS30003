System Architecture Summary

1. High-Level Design Overview

This system bridges a real-time 20ms audio stream across a flaky, hostile UDP network. To overcome severe packet drops and random delay spikes without the round-trip latency penalty of ARQ (Automatic Repeat reQuest), the architecture relies on a deterministic Forward Error Correction (FEC) mechanism, specifically a 2:1 XOR Parity strategy. This ensures recovery of dropped frames while strictly adhering to the ≤ 2.0x bandwidth limit.

2. Sender Architecture (System Level)

The sender reads raw 164-byte payload chunks from the harness source via a standard blocking UDP socket on port 47010. To implement FEC efficiently:

  • Packet Framing: The sender encapsulates the incoming sequence number (converted via ntohl/htonl for network byte order correctness) and payload into a custom 165-byte wire format. The first byte acts as a flag (0 for Data, 1 for Parity).
  • Memory Caching: A local 160-byte uint8_t array caches the previous frame's payload.
  • Bitwise XOR: On every even sequence number, a third parity packet is generated by performing a byte-by-byte XOR operation between the cached payload and the current payload. This yields an exact bandwidth overhead of 1.55x, well below the 2.0x threshold limit.

3. Receiver Architecture (System Level)

The receiver operates on port 47002 and is responsible for reconstructing the stream and enforcing strict playout deadlines.

  • Non-Blocking I/O: Standard recvfrom calls are blocking, which would cause the receiver to miss deadlines if the relay drops a packet. To solve this, the receiver uses a poll() system call with a minimal timeout. This allows the thread to continuously check the system clock and manage deadlines even when the network goes silent.
  • Jitter Buffer & Memory Management: Incoming packets are stored in a std::map structure acting as a dynamic jitter buffer. A separate map holds parity data. To prevent memory leaks during extended runs, packets that fall too far behind the current playout sequence are safely erased from the heap.
  • Instant Reconstruction: When a parity packet arrives, the receiver checks the data map. If a base packet is missing, it instantly applies a reverse XOR operation against the surviving packet to perfectly reconstruct the lost 160-byte payload on the fly.
  • Deadline Playout Loop: Instead of waiting for a packet to arrive, a while loop continuously checks current_time_ms() >= t0_ms + delay_ms + (seq * 20). If the deadline expires, the system immediately flushes the buffered packet to the harness player via port 47020, or skips the sequence if recovery was impossible, ensuring subsequent frames are not delayed.
